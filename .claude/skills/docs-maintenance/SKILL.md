# Documentation Maintenance

When asked to create or update documentation in the Vizcraft repository, follow these rules to ensure high‑quality, well‑structured docs:

## 1. Determine the document type

- **Identify the purpose of the requested content**. Use the Diátaxis framework to classify it as a **tutorial**, **how‑to guide**, **reference** or **explanation**:contentReference[oaicite:39]{index=39}.
- **Place the document in the correct folder** under `docs/` (`tutorials/`, `how-to/`, `reference/`, `explanations/`). Align the folder structure with the sidebar so that it can be autogenerated:contentReference[oaicite:40]{index=40}.
- If the request spans multiple types, split it into separate documents (e.g., a how‑to guide and a conceptual explanation) and cross‑link them.

## 2. Write with clarity and consistency

- Use **active voice** and **simple, short sentences**:contentReference[oaicite:41]{index=41}. Break up long paragraphs with headings or lists.
- Use `h2` and `h3` headings to structure content. These levels appear automatically in the table of contents:contentReference[oaicite:42]{index=42}.
- Present information as bullet or numbered lists when appropriate. Avoid embedding long blocks of prose in tables.
- Include **code snippets** for examples. Place examples in tutorials or how‑to guides, not in reference docs:contentReference[oaicite:43]{index=43}.
- Follow the project’s style guide (tone, capitalization, formatting) and update `custom.css` or the theme if global styles are needed.

## 3. Use front matter and tags

- Include a **descriptive `id`** (default is the file name) and **stable `slug`** for each document. Use the `slug` front matter to prevent broken links when renaming files:contentReference[oaicite:44]{index=44}.
- Add a short **`description`** to improve SEO and search results.
- Assign **tags** relevant to the document’s content. Predefine tags in `docs/tags.yml` for common categories (e.g., `installation`, `visualization`, `api`).

## 4. Update navigation

- Ensure that new documents appear in the sidebar. When using autogenerated sidebars, confirm the folder structure reflects the desired hierarchy. When using manual sidebars, update `sidebars.js` to include the new document under the correct category:contentReference[oaicite:45]{index=45}.
- Add or update **links** in the navbar or homepage if the document is important (e.g., a new tutorial).

## 5. Maintain versioning and archives

- When releasing a new version, run `docusaurus docs:version <version>` to snapshot the current docs. Commit the `versioned_docs` and `versioned_sidebars` directories and update the version dropdown in `docusaurus.config.js`:contentReference[oaicite:46]{index=46}.
- When changing or removing functionality, update the documentation for each supported version. Add a banner in old versions if the content is outdated.
- Archive deprecated documents under an `archive/` folder and remove them from the main sidebar and search index:contentReference[oaicite:47]{index=47}.

## 6. Review and test documentation changes

- **Run automated link and style checks**. Execute the markdown link checker and linter to catch broken links and style violations.
- **Test tutorials and examples**. Execute the code snippets in tutorials to ensure they still work. If a snippet fails, update the documentation or create an issue for code changes.
- **Seek feedback**. If the documentation lacks detail or context, ask the user or maintainer for clarification before proceeding. Do not rely solely on the issue title to understand requirements.

## 7. Commit and contribute

- Create a branch for the documentation changes. Follow the naming convention: `docs/<issue-number>-<short-description>`.
- Write descriptive commit messages following the project’s conventions (e.g., Conventional Commits). Include `docs:` at the start of the subject line when modifying documentation.
- Push the branch and open a pull request. Ensure the PR template is filled out, including checkboxes for documentation and tests.

## 8. Encourage community involvement

- At the bottom of each page, include a link to the GitHub issue tracker with a prefilled `documentation` label so users can report issues:contentReference[oaicite:48]{index=48}.
- Respond to feedback on documentation pull requests and incorporate suggestions.
- Keep documentation inclusive and accessible by following accessibility guidelines and avoiding assumptions about the reader’s background.

By following these rules, the AI assistant will maintain Vizcraft’s documentation in a structured, readable and user‑friendly manner.
